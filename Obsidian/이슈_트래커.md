### [LIVE-09] Watcher `_execute_buy` 중복 호출 (asyncio race)
- **발생일**: 2026-04-21 (Day 3)
- **종결일**: 2026-04-21
- **R-N / Phase**: LIVE 운영, W-LIVE-09
- **심각도**: 🔴 Critical (P0) → ✅ 해결완료
- **상태**: ✅ 해결완료

#### 현상
Day 3 10:49:59 삼성전기 009150 트리거 시 `[Coordinator] 매수 발주: 삼성전기` 로그 8회 / 0.22초 내 집중 발생. KIS 접수 3건 (order_id 0020351800/1900/2000, 3주 @ 753,000원), 나머지 5건 "주문가능금액 초과" 리젝. buy2 리젝 다수 ("초당 거래건수 초과" 포함).

#### 원인 확정
**(c) asyncio tick handler race.** `_process_signals` 에서 `await self._execute_buy(chosen, ts)` **전에** `self._active_code` 가 설정되지 않아, await 동안 다른 tick 이 재진입 시 `_active_code is None` → 같은 chosen watcher 로 중복 진입.

- (a) AES 복호화 실패(LIVE-08) → `trader.has_position()` False: **증폭 요인이었으나 근본 원인 아님**. LIVE-08 해결만으로는 재발 방지 불가.
- (b) Watcher 중복 바인딩: **기각**. 동일 Watcher 인스턴스의 `_execute_buy` 가 여러 번 호출된 구조.

#### 해결 (W-LIVE-09, commit `474d5a7`)

**3중 방어**:
1. **Coordinator `_process_signals` lock-first** (3 사이트: 첫 매수 / T3 예약 진입 / T3 재판정). `self._active_code = chosen.code` 를 `await _execute_buy` **이전**에 설정 → 다음 tick 재진입 차단. 실패 시 `except` 에서 active 해제 + raise.
2. **`_execute_buy` idempotency guard**. 같은 watcher 에 `buy1_placed=True` 로 진입 시 즉시 return.
3. **`Trader.place_buy_orders` 재진입 차단**. `self.pending_buy_orders` 중 `is_active` 존재 시 CRITICAL 로그 + return.

#### Day 4 실전 검증 포인트
- `[Coordinator] 매수 발주: {code}` 로그 1종목당 1회만 출력 (0.22초 내 8회 재발 금지)
- `[Trader] CRITICAL: place_buy_orders 재진입 차단` 로그 0건 (3차 방어선 자체가 작동하면 1/2차 보완 필요)

#### 관련
- `src/core/watcher.py::_process_signals` (L842-851, L1301-1312, L1341-1350)
- `src/core/watcher.py::_execute_buy` (L857-863)
- `src/core/trader.py::place_buy_orders` (L62-74)

### [LIVE-10] 미체결 매수 고아 주문 회수 누락
- **발생일**: 2026-04-21 (Day 3)
- **종결일**: 2026-04-21
- **R-N / Phase**: LIVE 운영, W-LIVE-10
- **심각도**: 🔴 High (P1) → ✅ 해결완료
- **상태**: ✅ 해결완료

#### 현상
Day 3 10:55:00 매수 마감 시 Trader `cancel_buy_orders` 가 tracked 1건만 취소 시도. LIVE-09 로 발생한 8건 발주 중 나머지 7건은 취소 경로에 진입조차 못 함.

- KIS 에러 `[TTTC0803U]: 정정취소 가능수량이 없습니다` (이미 체결 완료) — 에러 자체는 정상
- **진짜 문제**: Coordinator `pending_buy_orders` 가 8건 발주 중 1건만 보유. order_id → Watcher 매핑이 중복 발주 경로에서 누락

#### 원인
`trader.cancel_buy_orders` 가 **tracked-only** (`self.pending_buy_orders`). Watcher 밖에서 발생한 고아 주문 회수 불가.

#### 설계 원칙 (수석님 결정)
- `cancel_buy_orders` **실시간 타이밍 경로는 tracked-only 유지**. cancel 경로에 inquire API 삽입은 timing-critical 을 해친다. HTS 매매 개입 금지 정책과 유사.
- 대신 **시각 기반 독립 safety net** 으로 보강 (부팅 1회 + 10:55 1회).

#### 해결 (W-LIVE-10, commit `474d5a7`)
1. **`kis_api/constants.py`**: `TR_UNFILLED = "TTTC0084R"` (구 TTTC8036R) 추가. 공식 API 문서 (`Obsidian/한국투자증권_오픈API_전체문서_20260416_030007.xlsx`) 검증.
2. **`kis_api/kis.py`**: `inquire_unfilled_orders()` 메서드 신규. `INQR_DVSN_2="2"` (매수만 필터), `sll_buy_dvsn_cd="02"` + `psbl_qty > 0` 추가 필터.
3. **`main.py`**: F1 SA-5e 확장 — 부팅 직후 미체결 매수 조회 + 발견 시 전량 `cancel_order` + 텔레그램 critical 알림. "포지션 0 + 미체결 있음" 공백 커버 (holdings-기반 SA-5e 가 놓치던 시나리오).
4. **`watcher.py`**: `on_buy_deadline` 말미 10:55 일일 safety net. 같은 inquire → 전량 취소. 루프 재발 시 마지막 차단선.

#### Day 4 실전 검증 포인트
- 부팅 시 `[F1 확장] ... 미체결 매수 ...건 감지` INFO (발생 0건 = 정상)
- 10:55 `[Coordinator] 10:55 safety net:` INFO (발생 시 orphan 존재 = LIVE-09 재발 의심)

#### 관련
- `src/kis_api/constants.py:57`
- `src/kis_api/kis.py:762` (`inquire_unfilled_orders`)
- `src/main.py:244` (F1 SA-5e 확장)
- `src/core/watcher.py:1013+` (10:55 safety net)
- trader.cancel_buy_orders 는 의도적으로 tracked-only 유지

- **종결일**: 2026-04-17

- **상태**: ✅ 해결완료

```
10:07:18  [동결] buy1  24,990,750원
10:07:18  [동결] buy2  24,987,280원
10:07:18  [확정] buy1  24,990,750원  사용→24,990,750 ✅

10:10:12  [복원] 0원  P&L -24,990,750원  ← 버그!
```

`restore(0원)` 호출됨 = `exit_value = 0`

#### 원인

```python
exit_value = watcher.current_price * self.trader.position.total_qty if self.trader.position else 0
```

```python
exit_value = watcher.current_price * watcher.total_buy_qty if watcher.total_buy_qty > 0 else 0
```

- `trader.py:simulate_fills`에 디버그 로그 추가 (근본 원인 추적용)

`src/main.py`, `src/core/trader.py`
