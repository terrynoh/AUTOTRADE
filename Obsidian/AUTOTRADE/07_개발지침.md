## 규칙 1: 하드코딩 금지 — 모든 설정값은 구조화

전략 수치뿐 아니라 **인프라 설정, 타임아웃, 포트, 로그 보관 기간 등** 최적화 과정에서
지속적인 수정이 발생할 수 있는 모든 값은 반드시 설정 파일에서 로드한다.

### 설정 계층

```
config/strategy_params.yaml     ← 모든 파라미터 원본 (YAML)
config/settings.py              ← Pydantic 모델로 타입 검증 + 기본값
  ├── ScreeningParams           ← 스크리닝 조건
  ├── MultiTradeParams          ← 멀티 트레이드
  ├── EntryParams               ← 매수 진입
  ├── ExitParams                ← 청산 조건
  ├── OrderParams               ← 주문 설정
  ├── RiskParams                ← 리스크 관리
  ├── ApiParams                 ← API Rate Limit 등
  ├── MarketParams              ← 시장 시간 (장 시작/마감)
  └── InfraParams               ← 인프라 (타임아웃, 로깅, 대시보드)
```

### 하드코딩 허용 범위

| 허용 | 불허 |
|------|------|
| 진짜 상수 (API URL, TR ID, KRX 호가 단위 테이블) | 시간 값 ("09:50", "15:30" 등) |
| Pydantic 모델의 `default` 값 (= YAML 미지정 시 폴백) | 타임아웃/간격 (10, 30, 60초 등) |
| | 포트 번호, 버퍼 크기 |
| | 전략 관련 모든 수치 (%, 분, 원) |
| | 로그 보관 기간 |

### 코드에서 참조하는 방법

```python
# 잘못된 예 (하드코딩)
timeout = aiohttp.ClientTimeout(total=10, connect=5)
await self._wait_until(time(15, 30))
if age > 60 and self.api.ws_connected:

# 올바른 예 (설정 참조)
timeout = aiohttp.ClientTimeout(
    total=self._infra.http_timeout_total_sec,
    connect=self._infra.http_timeout_connect_sec,
)
close_time = time.fromisoformat(self.params.market.close_time)
await self._wait_until(close_time)
if age > infra.ws_timeout_sec and self.api.ws_connected:
```

### 새 파라미터 추가 절차

1. `config/strategy_params.yaml`에 키 추가
2. `config/settings.py`의 해당 Pydantic 모델에 필드 추가 (타입 + 범위 검증)
3. 코드에서 `self.params.xxx.yyy` 형태로 참조
4. 기존 하드코딩 값은 Pydantic 모델의 `default`로 보존 (YAML 미지정 시 호환)

## 규칙 4: 매매 모드 전환 순서

```
dry_run → paper → live
```

- `dry_run`: 주문 없이 시그널 로깅 + 완전한 가상 체결/P&L 추적

## 규칙 7: 대시보드 UI 수정 시 필수 체크리스트

백엔드(`app.py`) 또는 프론트엔드(`index.html`) 수정 시 반드시 아래 확인:

1. **숫자 포맷 함수** (`fmtWon`, `fmtPrice`) 수정 시 → `0`, `null`, `undefined`, 음수, 양수 5가지 테스트
2. **백엔드 응답 구조** (`to_dict()`) 변경 시 → `index.html`의 `updateUI(d)` 필드 참조 일치 확인
3. **설정 참조** 변경 시 → `self.params`가 `None`인 초기 상태에서 fallback 존재 여부 확인
4. **수정 후** → 반드시 브라우저에서 `/api/status` JSON 확인 + 대시보드 화면 확인
5. **대시보드-트레이더 연동**: `app.py`는 단독 실행 시에도 자동 연결(`@app.on_event("startup")`)

> ISSUE-003 참조: 코드 수정 → UI 깨짐 → 재수정 반복 방지 목적

## 규칙 8: 네트워크 서비스 수정 시 End-to-End 연결 검증

Tunnel, 프록시, 리버스 프록시 등 **네트워크 경로가 포함된 기능** 수정 시 반드시 아래 체크:

### 필수 체크리스트

1. **리슨 확인**: 목적지 포트에 서버가 실제로 리슨 중인가?
   ```bash
   netstat -an | findstr :8503
   curl http://localhost:8503
   ```
2. **프로세스 의존성**: 서비스 A가 서비스 B에 의존하면, A 시작 시 B도 함께 시작되는가?
   - 예: Tunnel → 대시보드 서버. Tunnel만 시작하고 서버를 안 띄우면 502
3. **포트 일치**: `.env`, `settings.py`, Tunnel 설정, `run_*.bat` 파일의 포트가 모두 동일한가?
4. **URL 갱신**: Cloudflare Quick Tunnel은 재시작 시 URL이 변경됨. 이전 URL 사용 불가

### 디버깅 순서 (502 발생 시)

```
1단계: curl http://localhost:<port>      → 서버 리슨 여부
2단계: netstat -an | findstr :<port>     → 포트 상태
3단계: 로그에서 "서버 시작" 메시지 확인   → 서버 기동 여부
4단계: 포트 번호 불일치 확인              → .env, settings, tunnel 설정 대조
```

**표면 증상(URL, 포트 번호)에 집중하지 말고, 근본 원인(서버가 떠 있는가?)부터 확인할 것.**

> ISSUE-005 참조: Tunnel은 열었지만 대시보드 서버를 안 띄워서 502 반복

## 규칙 9: 기능 수정 시 영향 경로 전체 추적

하나의 기능을 수정할 때 **입력 → 처리 → 출력** 전체 경로에서 영향받는 코드를 모두 찾아 수정할 것.

### 체크리스트

1. **처리 로직** 수정 시 → 해당 값을 **사용하는 곳** (UI 표시, 알림 발송, API 응답) 모두 확인
2. **인증/권한** 수정 시 → 인증 정보를 **전달하는 곳** (URL 생성, 링크 발송, 리다이렉트) 모두 확인
3. **데이터 구조** 변경 시 → 해당 데이터를 **읽는 곳** (프론트엔드, 다른 모듈, 테스트) 모두 확인

### 실제 사례

| 수정 내용 | 빠뜨린 것 | 결과 |
|-----------|-----------|------|
| `app.py`에 `?token=` 인증 추가 | `main.py` 텔레그램 URL 발송에 토큰 미포함 | 매번 토큰 없는 URL 발송 → 수동 재발송 반복 |
| `main.py`에 Tunnel 시작 추가 | 대시보드 서버(uvicorn) 시작 누락 | 502 반복 |

> ISSUE-006 참조: 처리만 고치고 출력을 빠뜨린 사례
